# Session ID Architecture

This document explains the various identifiers used to track agent sessions in Homespun, why each exists, and how they relate to each other.

## Overview

Homespun uses multiple identifiers to track agent sessions across different layers of the system. This is intentional - each ID serves a distinct purpose and cannot be consolidated without losing important functionality.

## Session ID Inventory

| ID Type | Generated By | Example | Purpose |
|---------|--------------|---------|---------|
| **Server Session ID** | Server (`Guid.NewGuid()`) | `a1b2c3d4-...` | Primary session identifier used throughout the application |
| **Worker Session ID** | Worker (`randomUUID()`) | `e5f6g7h8-...` | Internal identifier within the worker container |
| **Conversation ID** | Claude SDK | `i9j0k1l2-...` | Claude Code's session ID for resumption (`--resume` flag) |
| **Container ID** | Docker Engine | `sha256:abc...` | Docker container identifier |
| **Container Name** | Server (derived) | `homespun-issue-{projectId}-{issueId}` | Human-readable container name for reuse detection |
| **Entity ID** | External (Fleece/GitHub) | `abc123` | Issue or PR identifier |
| **Project ID** | External | `my-project` | Project identifier |

## Why Multiple IDs Are Necessary

### Can we just use Container ID and Claude Code session ID?

**No.** Here's why:

1. **Container ID** is a Docker runtime artifact (long SHA256 hash):
   - Not human-readable
   - Changes if container is recreated
   - Not known until container starts

2. **Claude Code session ID (Conversation ID)** is only available after SDK initialization:
   - We need a session ID immediately when the request arrives
   - The SDK returns its ID in the first response message
   - Used only for session resumption (\`--resume\` flag)

3. **Entity IDs (ProjectId, IssueId)** are required for:
   - Business logic routing
   - UI navigation
   - Container reuse detection (via derived container name)

### ID Relationships

\`\`\`
Server Session ID ─────┬──── Used by: API endpoints, SignalR groups, UI routing
                       │
                       ├──── Contains reference to: Conversation ID (for resume)
                       │
                       └──── Maps to: Container (via tracking dictionary)

Container Name ────────┬──── Derived from: ProjectId + IssueId
                       │
                       └──── Enables: Container reuse across sessions

Conversation ID ───────┬──── Provided by: Claude SDK
                       │
                       └──── Used for: Session resumption (--resume flag)
\`\`\`

## Key Implementation Details

### Server Session ID Generation

Location: \`src/Homespun.Server/Features/ClaudeCode/Services/DockerAgentExecutionService.cs\`

\`\`\`csharp
var sessionId = Guid.NewGuid().ToString();
\`\`\`

This ID is generated immediately when a session request arrives and is used throughout the server to track the session.

### Worker Session ID Generation

Location: \`src/Homespun.Worker/src/services/session-manager.ts\`

\`\`\`typescript
const id = randomUUID();
\`\`\`

The worker generates its own internal ID for session tracking. The server then remaps all SDK messages to use the Server Session ID via \`RemapSessionId()\`.

### Conversation ID Capture

The Conversation ID is captured from the Claude SDK's first response message:

\`\`\`typescript
// Worker captures it from SDK messages
if (msg.session_id) {
  ws.conversationId = msg.session_id;
}
\`\`\`

This ID is stored and used when resuming sessions with the \`--resume\` flag.

### Container Naming Convention

Location: \`src/Homespun.Server/Features/ClaudeCode/Services/DockerAgentExecutionService.cs\`

\`\`\`csharp
internal static string GetIssueContainerName(string projectId, string issueId)
    => \$"homespun-issue-{projectId}-{issueId}";
\`\`\`

This deterministic naming enables container reuse detection - when a new session is requested for the same issue, the existing container can be reused instead of starting a new one.

## Summary

The current session ID architecture is intentionally layered:

- **Server Session ID**: Immediate, application-wide session tracking
- **Worker Session ID**: Internal worker process tracking (remapped to Server ID)
- **Conversation ID**: Claude SDK resumption capability
- **Container ID/Name**: Docker resource management and reuse

Each layer serves a distinct purpose that cannot be eliminated without losing functionality.

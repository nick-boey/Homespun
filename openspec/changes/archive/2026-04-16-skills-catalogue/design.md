## Context

Prompts today live in a two-tier DB catalogue with `{{placeholder}}` + `{{#if}}` templating, 13 seeded globals and per-project overrides, rendered into an `initialMessage` on session start. This change replaces that system with filesystem skills under `.claude/skills/**/SKILL.md`.

OpenSpec installs 8 skills, each with frontmatter `metadata.author: openspec`. These are generic: they call `openspec status` and `openspec instructions` at runtime, so custom schemas change what they do (which artifacts they iterate), not which skills exist. Since Claude Code commands are being deprecated in favour of skills, skills are the sole dispatch mechanism.

Experimental findings (2026-04-16):
- `openspec update --force` regenerates `SKILL.md` files, stripping any custom frontmatter.
- Sibling sidecar files survive `openspec update --force` (confirmed for both skills and commands directories).
- Custom schemas do not generate new skills — fixed set of 8 regardless of active schema.
- All 8 OpenSpec skills carry `metadata.author: "openspec"` in frontmatter.

## Goals / Non-Goals

**Goals:**
- Skills as the dispatch contract: app reads SKILL.md body + args → sends as initialMessage.
- CLI parity: same skill body, same behaviour whether invoked from CLI or app.
- OpenSpec skills discovered by frontmatter (`metadata.author == "openspec"`).
- Custom Homespun prompt skills discovered by `homespun: true` in SKILL.md frontmatter.
- Survive `openspec update --force` (only relevant for OpenSpec-generated skills, which are hard-coded anyway).

**Non-Goals:**
- Template engine (deleted).
- Command-based dispatch (commands deprecated in Claude Code).
- Global skills — skills are repo-scoped.
- Per-user skill overrides.

## Decisions

### D1: Skills as sole dispatch mechanism

**Decision:** The app reads the skill's `SKILL.md` body, appends composed args (e.g., change name, issue id), and sends the combined text as `session.initialMessage`. No slash-command invocation syntax.

**Rationale:** Commands are being deprecated from Claude Code in favour of skills. Skills are the forward-looking mechanism. The skill body IS the prompt — the app just reads a file instead of querying a DB row. In the CLI, Claude auto-invokes the same skill via description matching, achieving parity.

### D2: Two discovery paths, one service

**Decision:** `SkillDiscoveryService` scans `.claude/skills/**/SKILL.md` and returns:
- **OpenSpec skills**: hard-coded list of 8 skill names. Not dynamically discovered — Homespun knows them by name. Surfaced in the OpenSpec tab.
- **Homespun prompt skills**: matched by `homespun: true` in SKILL.md frontmatter. These are user-authored and not regenerated by OpenSpec. Surfaced in the Task Agent tab as selectable prompts.
- **All other skills**: surfaced in the session chat window for Claude to auto-invoke during freeform conversation.

**Rationale:** Single scan, three uses. OpenSpec skills are hard-coded because their set is fixed and stable — dynamic detection via `metadata.author` is unnecessary complexity. Homespun skills use in-file frontmatter because they are user-authored and not subject to regeneration by `openspec update`.

### D3: Homespun skill metadata in SKILL.md frontmatter

**Decision:** Custom Homespun prompt skills declare their metadata directly in SKILL.md frontmatter:

```yaml
# .claude/skills/fix-bug/SKILL.md frontmatter
---
name: fix-bug
description: Fix a bug described in a Fleece issue
homespun: true
homespun-mode: build
homespun-args:
  - name: issue-id
    kind: issue           # UI shows issue picker
---
```

No sidecar file needed. OpenSpec skills are hard-coded (D7) so their metadata is not derived from frontmatter or sidecars.

**Rationale:** Homespun skills are user-authored and not regenerated by `openspec update`. Frontmatter is the natural place for metadata — simpler than a sidecar, one file per skill. The sidecar concern only applied to OpenSpec-generated skills where frontmatter is overwritten, but those are hard-coded anyway.

### D4: Args schema with `kind` enum

**Decision:** Skills declare args in their frontmatter under `homespun-args` with a `kind` field from a small enum: `issue`, `change`, `phase-list`, `free-text`. The UI builds the right input control per kind.

**Rationale:** Enables the UI to build inputs per skill without special-casing. The frontmatter carries what the skill needs; the UI reads it.

### D5: Skill propagation mirrors `.fleece/` behaviour

**Decision:** The "main clone" holds canonical skills. New branch clones copy `.claude/skills/` at creation. In-flight branches use their own committed copy. Branch deletion + recreation is the path for picking up fixes.

**Rationale:** Consistent with git semantics and how Claude Code resolves repo-scoped skills.

### D6: Keep `MiniPromptService`

**Decision:** Out of scope for deletion. Generates branch-name slugs, not prompts.

### D7: OpenSpec skills hard-coded in Homespun

**Decision:** The 8 OpenSpec skill names are hard-coded in the server and UI:

| Skill name | Purpose | Default when |
|---|---|---|
| `openspec-explore` | Thinking partner | No change or artifacts incomplete |
| `openspec-new-change` | Scaffold change dir | (manual, not auto-selected) |
| `openspec-propose` | Create all artifacts | (manual, alternative to explore→continue) |
| `openspec-continue-change` | Next artifact | (manual, for incremental artifact creation) |
| `openspec-apply-change` | Implement tasks | All artifacts created |
| `openspec-verify-change` | Verify impl matches specs | (manual, before archive) |
| `openspec-sync-specs` | Sync delta specs to main | (manual, utility) |
| `openspec-archive-change` | Archive completed change | All tasks complete |

Auto-selection logic:
1. No change or artifacts incomplete → default `explore`
2. All schema artifacts created → default `apply`
3. All tasks checked in tasks.md → default `archive`

**Rationale:** The skill set is stable across schemas (confirmed 2026-04-16). Hard-coding avoids dynamic discovery complexity and lets the UI provide purpose-specific UX per skill.

## Risks / Trade-offs

- **[R1]** Losing the bulk JSON prompt editor. Acceptable trade for CLI parity.
- **[R2]** Skills are not hot-reloaded into active sessions. Matches prior prompt behaviour.
- **[R3]** Task Agent tab retains the ability to dispatch an agent with no skill (empty prompt) or with a custom free-text message. This is the existing "start agent on issue" flow — user can paste the issue ID into the prompt if desired.
- **[R4]** If OpenSpec changes its skill names, the hard-coded list breaks. Mitigated by also matching `openspec-*` name prefix as fallback.

## Resolved Questions

- **OQ1 (resolved)**: Issues Agent tab remains skill-less.
- **OQ2 (deferred)**: Skill authoring UX tracked as follow-up Fleece issue.
- **OQ3 (resolved)**: No generic `/task` skill. Task Agent dispatches with no message by default; user types a custom message if desired.
